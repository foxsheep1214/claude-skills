## §3 三大痛点根因分析 编写规范

### 格式模板

```markdown
### 3.X 痛点标题

**根因一（简洁概括）**：展开分析内容。

**根因二（简洁概括）**：展开分析内容。

**根因三（简洁概括）**：展开分析内容。
```

### 三条根因的递进设计原则

- 根因一：问题的物理本质（为什么发生了）
- 根因二：为什么常规手段无法解决（从工程范式分析，如波束/距离/速度三维）
- 根因三：更恶劣的极端场景（如何进一步恶化）

**反面教材**：不要将三根因写成并列的"A/B/C"三个困难（如"看不见/看见太多/慢"），
那样的结构缺乏深度，应找到统一的核心（如"地杂波压倒性背景"），
三条根因都是这个核心的展开。

### 已确立的写法要点（2026-05-23迭代多轮后的经验）

1. **根因要有量化举例**：不要只说"RCS差距大"，要给出具体数字——山体/大楼RCS可达
   100,000㎡以上，无人机0.01㎡量级，差约70dB。

2. **根因要落在雷达工程机制上**：不要停留在现象描述。比如RCS鸿沟要解释"为什么雷达
   处理不掉"——带宽/脉宽/积累时间/波束宽度的限制导致强目标扩展掩盖小目标。

3. **根因二采用"三维分析框架"**（针对探测类痛点）：从波束（角度）、距离、速度三个
   维度逐一分析分辨力不足，每个维度都有具体的工程限制（波束副瓣泄漏、距离旁瓣、
   零速通道杂波残余）。

4. **根因之间的逻辑递进比编号顺序重要**：用户会调整顺序。最终确认的逻辑链是：
   "差距有多大"→"为什么分不开"→"为什么更糟"。

### §3.2 识别痛点根因

三条根因迭代确认（2026-05-23）：

1. **根因一：雷达可用的识别特征只有三个维度（RCS起伏、微多普勒、运动学），且均存在目标间重叠**
   - RCS随姿态角剧烈波动，单次采样不可靠
   - 微多普勒是旋翼类核心特征，但固定翼（Shahed）无此特征=零可用信息
   - 运动学特征需要较长跟踪时间，类间边界模糊

2. **根因二：识别架构受制于传统信号处理流水线（脉压→MTD→波束合成→CFAR→点迹凝聚→跟踪滤波→识别）**
   - CFAR检测丢弃低于门限的回波，微弱目标在第一级已丢失
   - 凝聚压缩丢弃RCS空间分布和微多普勒频谱细节
   - 滤波平滑抹除瞬时运动特征
   - 虚警点迹混入识别模块
   - 识别模块只有"二手数据"——概念是用户原创，不能丢失

3. **根因三：方法论仍以规则驱动为主，模型和数据积累严重不足**
   - 与光学识别存在代际差距
   - 规则体系无法处理类间混叠，无法从数据学习
   - 研究总量和数据量呈数量级差距
   - 训练数据采集成本远高于光学图像，公开数据集几乎不存在

## 关键技术知识点（本会话2026-05-23积累）

### X波段相干积累时间计算

- X波段（10GHz），λ=0.03m
- 1m/s速度多普勒频移：f_d = 2v/λ ≈ 67Hz
- 理论最小T_cpi = 1/67 ≈ 15ms
- 工程实际：杂波谱展宽需3-5倍，50-100ms

### 旋翼调制周期

- 典型四旋翼转速5000-10000RPM
- 2叶桨调制频率80-160Hz
- 叶片旋转周期6-12ms（非≥100ms）

### 雷达三维分辨维度

- 波束（角度）：波束副瓣泄漏、低空目标与强散射体同波束
- 距离：带宽决定分辨力、距离旁瓣扩展
- 速度（多普勒）：零速通道杂波残余、相干积累时间限制

### 警戒雷达与火控的根本矛盾（§3.3，2026-05-23确立）

**核心公式**（来自wiki知识库）：
- 帧周期约束：T_f = T_s + T_t（毕增军）
- 角精度公式：σ_θ ≈ λ/(2πd·cosθ·√(2·SNR))（Barton/张光义）
- TWS模式数据率上限 = 搜索帧率（0.5-1Hz），火控需要4-20Hz
- TAS模式：跟踪时间与搜索时间竞争（张光义）

**三条根因递进逻辑**：
1. **零和竞争**：T_f=T_s+T_t，T_t增加=搜索覆盖收缩
2. **波束体制矛盾**：警戒宽波束快覆盖 vs 火控窄波束高精度，同一天线孔径不可兼得
3. **TWS硬约束**：跟踪数据率受搜索帧周期限制，0.5-1Hz vs 4-20Hz，十倍缺口

**来源**：
- 毕增军《相控阵雷达资源管理技术》
- 张光义《相控阵雷达原理》
- Barton《Radar System Analysis and Modeling》

### 传统雷达流水线信息损耗链

脉压→MTD→波束合成→CFAR（信息第一次裁切）→点迹凝聚（信息压缩）→跟踪滤波（信息平滑）→识别（只能基于二手数据）

### patch工具处理含重复章节标题的大文档时容易失败

**症状**：当文档中存在重复的章节编号（如同时有新旧两套§4/5/6），
patch的old_string匹配会因内容过长而失败。

**解法**：用execute_code + Python直接操作字符串，通过
`content.find("唯一标记")`定位边界，`content[a:b] + content[c:d]`
拼接，避免逐段patch。

**典型代码模式**：
```python
from pathlib import Path
path = Path("文档路径")
content = path.read_text()
dup_start = content.find("重复内容起点标记")
sec7_start = content.find("下游保留内容起点标记")
new = content[:dup_start] + content[sec7_start:]
path.write_text(new)
```
