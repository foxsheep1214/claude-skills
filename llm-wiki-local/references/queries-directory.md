# queries/ directory triage

The `wiki/queries/` directory is LLM Wiki's catch-all for **transient work artifacts**. It's NOT cleaned automatically and accumulates. Expect **5 categories of file** after a few weeks of activity.

## The 5 categories

| # | Category | Source | Format | Disposition |
|---|---|---|---|---|
| 1 | **PDF-text-recovery query** | Manually created when a PDF extracts to empty text (image-only scan) | `*.md`, no `origin:` field, `tags: [PDF文本恢复, OCR, ...]`, 2-5 KB | **Keep** — these are the user's standing work orders for OCR/re-ingest. Mark status in body. **See `pdf-recovery-workflow.md` for the end-to-end "do real work" workflow.** |
| 2 | **Deep-research report** | LLM Wiki deep-research pipeline, OR generated by user's `/research` slash command | `research-<topic>-<timestamp>.md`, has `origin: deep-research`, 10-30 KB, fully cited | **Keep** (high-value) or **delete** (low-value, see §3) |
| 3 | **Placeholder research (junk)** | Deep-research triggered by an **empty/placeholder prompt** like "完成这个query" or "test" | `research-完成这个query-...md`, `research-测试-...md`, content is unrelated noise | **Delete** (see §3) |
| 4 | **Short user-driven question** | User typed a question into the chat that didn't get answered | `<topic>-<timestamp>.md`, no `origin:`, 0.5-2 KB, just a "问题陈述" paragraph, no answer | **Answer it** (route to a real research) or **delete** |
| 5 | **Duplicate research** | Two deep-researches for the same topic (one user-driven, one auto) | `research-<X>-<ts1>.md` + `<X>.md` (no research- prefix) | **Mark superseded, don't delete.** Add `superseded_by: <canonical-file>` to the derivative's frontmatter, plus a banner pointing to the canonical file. Apply the same to the mirror copy in `wiki/sources/` if LLM Wiki wrote research output there too (it sometimes does — the sources/ copy is mis-classified as a source file but is actually a research process artifact). |

## How to identify each

**Quick classifier (one pass per file)**:

```python
import re
from pathlib import Path
from collections import defaultdict

queries = Path('<project>/wiki/queries')
cats = defaultdict(list)

for f in sorted(queries.glob('*.md')):
    text = f.read_text()
    fm_match = re.match(r'^---\n(.*?)\n---', text, re.DOTALL)
    fm = fm_match.group(1) if fm_match else ''
    size = f.stat().st_size
    has_origin = 'origin: deep-research' in fm
    has_research_prefix = f.name.startswith('research-')
    body_len = len(text) - len(fm_match.group(0)) if fm_match else len(text)

    if not has_origin and 'PDF文本恢复' in text or 'OCR' in fm and 'PDF' in text:
        cats['1_pdf_recovery'].append(f.name)
    elif has_origin and has_research_prefix and size > 5000:
        cats['2_research'].append(f.name)
    elif has_origin and has_research_prefix and '完成这个query' in f.name or '测试' in f.name:
        cats['3_junk'].append(f.name)
    elif not has_origin and size < 3000 and body_len < 1500:
        cats['4_short_q'].append(f.name)
    else:
        cats['5_duplicate_or_other'].append(f.name)

for k, vs in cats.items():
    print(f'{k}: {len(vs)}')
    for v in vs[:5]:
        print(f'  {v}')
```

**Manual signal-checking** (the script is heuristic; verify with these):

- **Category 1 (PDF-recovery)**: filename doesn't start with `research-`, body contains "## 问题陈述" + a list of OCR/方案 options. Almost always `tags: [OCR, PDF文本恢复, ...]`. For "how do I actually process one", see `pdf-recovery-workflow.md`.
- **Category 2 (real research)**: `origin: deep-research` in frontmatter, body has `## 概述` + numbered sections + `## 参考来源` with N inline `[1][2][3]` references. Often > 100 lines.
- **Category 3 (junk research)**: `origin: deep-research` but content has zero references to the wiki's actual topic (e.g. radar wiki has a research on Google autocomplete, Power Query, EVUS — none radar-related). Filename is a placeholder phrase ("完成这个query", "test", etc.).
- **Category 4 (short Q)**: no `origin:`, < 2 KB, body is one paragraph starting with "X 是什么" or "X 与 Y 的区别" — an unanswered question.
- **Category 5 (duplicate)**: filename with and without `research-` prefix, same topic, two timestamps. Check `related:` in frontmatter — if one points to the other, the non-`research-` one is the source-of-truth and the `research-` is the derivative. **Disposition**: add `superseded_by` + banner to the derivative, do not delete.

## Disposition matrix

| Category | Disposition | Backup before delete? |
|---|---|---|
| 1. PDF-recovery | **Keep.** These are long-lived work orders. Update `## 当前状态` in body when you act on them. **End-to-end workflow in `pdf-recovery-workflow.md`.** | No |
| 2. Real research (high-value) | **Keep.** Optional: merge into a related entity/concept page if the research has dense factual content (e.g. `research-黄培康-...` → `entities/黄培康.md`). Keep as standalone if it's a multi-source synthesis. | No |
| 2. Real research (low-value, off-topic) | **Keep** if it might be useful someday. **Delete** if you're sure. (The user may have asked about a tangentially-related topic.) | Yes |
| 3. Junk research | **Delete.** Back up to `~/.hermes/backups/wiki-queries-junk/<ts>/` first. These are noise. | **Yes** |
| 4. Short Q (unanswered) | **Answer it** if you have the source material; **delete** if the user moved on. | No |
| 5. Duplicate | **Mark superseded, don't delete.** Add `superseded_by: <canonical-file>` to the derivative's frontmatter, plus a banner pointing to the canonical file. Apply the same to the mirror copy in `wiki/sources/` if LLM Wiki wrote research output there too (it sometimes does — the sources/ copy is mis-classified as a source file but is actually a research process artifact). | No |

## The 5 PDF-recovery queries are the real cost

When category 1 accumulates (5+ files, all "未启动" or "进行中"), this is **the real cost** of using LLM Wiki on scanned/image-only PDFs. The fix is **not** to write more query files — it's to:

1. **Backup the original PDF** to `<project>/.llm-wiki/_pdf_split_backup/<ts>/`
2. **Split with pypdf** if the PDF is over the OCR engine's page limit (e.g. OpenXLab minerU is hard-capped at **200 pages/task** — undocumented error -60006; local minerU VLM has no page limit but is slow)
3. **Re-ingest** each part with sources/rescan after OCR completes
4. **Mark the query file's status** to "已完成 — 见 wiki/sources/<book>.md" when done

**Don't keep adding to `queries/` without acting on it** — the directory becomes a procrastination graveyard. The user's instinct to triage this is the right move.

### Batch systemic pattern (NEW 2026-06-10)

If you have N PDFs in the same "scanned Chinese book" failure mode, they likely share:

- Same producer string in PDF metadata (e.g. `producer: "IRIS"`)
- Same `_pdf_split/` Parts already prepared
- Same `minerU VLM` queued status — partial completion common (e.g. only `bjz_part01` ran vlm, 4 others not started)

When triaging one of them, build a **cross-batch table** in the query body so a future batch minerU job is scoped correctly:

| Book | Pages | Parts | minerU status |
|------|------|------|----------------|
| ... | ... | ... | ... |

This avoids re-discovering the systemic nature on the next query.

## Anti-patterns

- ❌ **Mass-deleting `queries/` to "clean up"** — you'll lose real work orders (PDF recovery).
- ❌ **Treating `research-` prefix as junk marker** — it's the opposite. `research-` is the GOOD content; the junk has the same prefix but junk content.
- ❌ **Merging real research into entity pages** without keeping a stub in queries/ — the original prompt context is lost.
- ❌ **Creating a category 4 short-Q and never answering it** — write the answer or delete the file.
- ❌ **Forgetting to update `## 当前状态` in PDF-recovery queries** when the OCR actually finishes.
- ❌ **Deleting Category 5 duplicates instead of marking superseded_by** — you lose the original prompt context. (NEW)
- ❌ **Trusting the query's "facts" without OCR-verifying** — they were often written by an LLM with no text to work from. Verify with PyMuPDF + Tesseract before propagating to entity/concept pages. (NEW)
- ❌ **OCR'ing the whole book in one shot** for metadata purposes — sample 8-12 strategically-chosen pages (cover/preface/TOC/first-body) is enough. (NEW)
